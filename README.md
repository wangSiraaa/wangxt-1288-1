# 环卫公厕巡检系统

## 环境要求
- **Node.js** >= 18.0.0
- **MySQL** >= 8.0 (推荐 8.0+)
- **npm** >= 9.0

---

## 快速启动（3步即可运行）

### 第 1 步：初始化数据库

在 MySQL 中执行 schema.sql，创建数据库、表结构，并初始化测试账号 `admin/123456`：

```bash
cd database
mysql -u root -p < schema.sql
```

> ⚠️ 若你的 MySQL root 账号有密码，请在 `-p` 后输入密码，例如：`mysql -u root -proot < schema.sql`

---

### 第 2 步：启动后端（端口 3001）

```bash
cd backend

# 1. 拷贝环境变量配置文件
cp .env.example .env

# 2. 修改 .env 中的数据库连接信息
#    编辑 MYSQL_HOST/MYSQL_PORT/MYSQL_USER/MYSQL_PASSWORD/MYSQL_DATABASE

# 3. 安装依赖
npm install

# 4. 启动服务（开发模式，支持热重启）
npm run dev
```

启动成功后会显示：
```
Server running on port 3001
Database connected successfully
```

#### .env 配置说明
| 变量名 | 说明 | 默认值 |
|--------|------|--------|
| PORT | 后端服务端口 | 3001 |
| MYSQL_HOST | MySQL 主机地址 | localhost |
| MYSQL_PORT | MySQL 端口 | 3306 |
| MYSQL_USER | MySQL 用户名 | root |
| MYSQL_PASSWORD | MySQL 密码 | 空 |
| MYSQL_DATABASE | 数据库名 | toilet_inspection |
| DB_CONNECTION_LIMIT | 连接池大小 | 10 |
| JWT_SECRET | Token 签名密钥 | 随机字符串 |
| JWT_EXPIRES_IN | Token 有效期 | 24h |

---

### 第 3 步：启动前端（端口 5173）

```bash
cd frontend

# 1. 安装依赖
npm install

# 2. 启动开发服务器
npm run dev
```

启动成功后浏览器访问 **http://localhost:5173**

## 业务流程验收指南

### 流程 1：保洁排班 → 打卡上传现场状态

**操作步骤：**
1. 登录 `admin/123456`，打开「保洁排班」
2. 点击「新增排班」，选择公厕、保洁员（cleaner01）、排班日期、班次时间段
3. 切换登录 `cleaner01/123456`，打开「打卡记录」
4. 点击「打卡」，选择刚创建的排班
5. 填写：打卡类型（上班打卡）、清洁评分、备注
6. **勾选**「厕纸不足」或「洗手液不足」（模拟现场状态）
7. 点击「提交打卡」

**预期结果：**
- ✅ 打卡成功，打卡列表出现新记录
- ✅ 排班状态自动变为「已完成」
- ✅ 由于勾选了耗材不足，**系统自动生成了耗材补给单**（查看「耗材补给」页面）

---

### 流程 2：低库存 → 自动生成补给单 → 派发 → 完成回写库存

**操作步骤（续流程1）：**
1. 切换登录 `supervisor01/123456`
2. 打开「耗材补给」，看到状态为「待处理」的自动生成补给单
3. 点击「派发」，选择执行人，确认
4. 执行人点击「完成补给」，填写实际补给数量

**预期结果：**
- ✅ 补给单状态按顺序流转：待处理 → 已派发 → 已完成
- ✅ **公厕的厕纸/洗手液库存数量自动增加**（打开「公厕管理」查看）
- ✅ 首页「耗材库存预警」数量减少

---

### 流程 3：设备故障上报 → 主管派发维修 → 维修完成

**操作步骤：**
1. 登录 `cleaner01/123456`，打开「设备维修」
2. 点击「上报故障」，选择公厕、故障类型、故障级别（紧急）、填写故障描述和照片
3. 切换登录 `supervisor01/123456`，打开「设备维修」
4. 点击「派发」，选择维修人员
5. 维修人员点击「开始维修」
6. 维修完成后点击「完成维修」，填写维修结果

**预期结果：**
- ✅ 工单状态按顺序流转：待处理 → 已派发 → 维修中 → 已完成
- ✅ 主管派发操作需要 supervisor/admin 角色，hotline 和 cleaner 无权限
- ✅ 维修列表可按故障级别筛选，紧急工单为红色标签

---

### 流程 4：市民投诉 → 分派处理 → 复查验证 → 关闭

**操作步骤：**
1. 登录 `hotline01/123456`，打开「投诉处理」
2. 点击「录入投诉」，填写投诉标题、类型（清洁/设备/人员）、优先级、内容、投诉人信息
3. 切换登录 `supervisor01/123456`，点击「分派」，选择处理人
4. 处理人登录后，在详情页提交处理方案
5. 登录任意有复查权限账号，点击「提交复查」：
   - 复查结果：选择 **「通过」**
   - 复查内容：填写问题确实已解决
   - 可上传照片、定位坐标
6. **尝试直接「关闭投诉」（无复查时）**

**预期结果（关键约束）：**
- ❌ 第6步会失败，提示 **「投诉关闭前必须至少关联一次复查记录」**
- ✅ 提交一条「通过」的复查后，再点击「关闭投诉」成功
- ✅ 投诉状态按顺序流转：待处理 → 处理中 → 待复查 → 已关闭
- ✅ 详情页「复查历史」按时间线展示所有复查记录（通过为绿色，不通过为红色）

---

### 流程 5：高峰时段缺岗自动预警

**触发条件：**
- 当前时间在公厕配置的 `高峰时段` 内（默认 07:00-09:00，可在公厕编辑中修改）
- 排班开始时间已过 **15 分钟** 但保洁员仍未打卡

**验证方法：**
1. 创建一个高峰时段内的排班（例如当前时间在 7:00-9:00，排班 7:00 开始）
2. 等待系统检查（每分钟自动运行一次，也可在「预警管理」点击「立即检查」）
3. 打开「预警管理」或首页看板

**预期结果：**
- ✅ 预警列表生成一条「缺岗预警」记录，级别为 high
- ✅ 顶部导航栏预警徽章显示未处理数量
- ✅ 处理人可以「处理」（标记已处理）或「忽略」预警

## 技术架构概览

### 后端技术栈
- **框架**：Koa 2.x
- **路由**：koa-router
- **数据库**：MySQL 8 + mysql2 Promise 连接池
- **认证**：JWT (jsonwebtoken) + bcryptjs 密码加密
- **定时任务**：60秒轮询（app.js 中 setInterval）
- **环境变量**：dotenv

### 前端技术栈
- **框架**：React 18 + TypeScript 5
- **组件库**：Ant Design 5.x
- **构建工具**：Vite 5
- **路由**：React Router 6
- **HTTP请求**：Axios（全局拦截、错误统一处理）
- **数据可视化**：ECharts 5

### 数据库表（共 10 张）
| 表名 | 说明 |
|------|------|
| users | 用户账号（admin/supervisor/hotline/cleaner 四角色） |
| areas | 片区管理 |
| toilets | 公厕基础档案（含厕纸/洗手液库存） |
| schedules | 保洁排班表 |
| check_ins | 打卡记录（含现场照片、评分、耗材状态） |
| supply_orders | 耗材补给单（自动+手动） |
| repair_orders | 设备维修工单 |
| complaints | 市民投诉记录 |
| complaint_reviews | 投诉复查记录（多对一关联投诉） |
| alerts | 预警记录（缺岗/低库存/投诉超时） |

## 常见问题排查

### 后端启动报 Database connection failed
1. 检查 MySQL 服务是否已启动
2. 检查 `.env` 中的 `MYSQL_HOST/PORT/USER/PASSWORD` 是否正确
3. 确认数据库 `toilet_inspection` 已创建（schema.sql 会自动创建）

### 前端登录提示 401 Unauthorized
1. 确认后端 `http://localhost:3001` 正常运行（浏览器访问 http://localhost:3001/api/auth/login 应返回 404 页面而非连接错误）
2. 确认 Vite 代理配置 `vite.config.ts` 中 `target` 指向正确的后端地址

### 低库存未自动生成补给单
1. 检查公厕的 `厕纸库存 <= 厕纸阈值` 或 `洗手液库存 <= 洗手液阈值`（可在「公厕管理-编辑」中设置）
2. 尝试在「耗材补给」页面点击「库存检查」按钮手动触发
3. 打卡时勾选「厕纸不足」或「洗手液不足」也会立即触发检查

### 无法关闭投诉（提示需要复查记录）
**这是预期的业务约束**，确保投诉确实解决才能关闭：
1. 进入投诉详情 → 点击「提交复查」
2. 复查结果选择「通过」→ 提交
3. 再点击「关闭投诉」即可成功
